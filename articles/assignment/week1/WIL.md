RESTful API, DTO에 대해

REST API는 웹 상에서 컴퓨터 시스템 간 정보 교환을 위한 인터페이스, 통신 규약이다. HTTP 프로토콜을 사용하는데, HTTP가 기본 행위를 정의한다면 REST는 좀 더 구조화된 규칙을 제시한다.  REST는 Uniform Stateless, Cacheable, Self-descriptiveness, Client-Server 구조, 계층형 구조의 특징을 가진다. 이 중 Client-Server 구조에 의해 클라이언트와 서버 간 의존성이 줄어들게 된다. 또한 REST는 무상태성 성격을 갖는다. 서버는 클라이언트의 상태 정보를 저장하지 않는다는 것이다. 클라이언트가 서버로 요청을 보낼 때, 해당 요청을 처리하는데 필요한 모든 정보를 요청 메시지 자체에 담아서 전송해야 한다. 쇼핑몰 사이트에서 물건을 구매하는 경우, 고객의 로그인 정보와 장바구니의 상품 정보를 모두 '구매 요청'에 담아야한다는 것이다. 서버가 클라이언트 상태를 저장하지 않기 때문에 서버를 필요에 따라 쉽게 추가하거나 제거할 수 있어 대규모 트래픽을 처리하는데 용이하다. 또한 서버가 다운되어도 상태정보가 서버에 묶여 있지 않기 때문에, 세션 정보가 유실될 위험이 적다.

DTO는 프로세스 간 데이터를 전달하는 객체이다. C언어의 구조체와 유사하게 전달하고 싶은 데이터를 묶어서 하나의 단위로 다룬다. 과제를 하며 요청, 응답의 DTO를 하나하나 만드는 대신 기존의 엔티티를 사용하면 더 쉽고 빠르지 않나하는 생각이 들었다. 하지만 이 경우 DTO만 사용했을때 보다 더 큰 데이터 크기가 요구되면서, 오버헤드가 발생할 수 있다. 또한 엔티티에 변경사항이 생기면 엔티티를 사용한 API에도 영향이 가기 때문에 DTO를 사용하는 것이 관리에 좋다.
