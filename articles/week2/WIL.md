# Service 계층

## 계층 구조  : Layered Architecture
관심사 별로 여러개의 계층 분리  
+ 단방향 의존성 : 하위는 상위를 몰라야 한다

+ 장점
    + 독립적인 모듈 개발, 재사용성 증가
    + 코드의 단순화 , 유지보수 용이
    
+ 단점 
    + 오버헤드 : 20-80규칙 20은 단순 통과처리 80은 비즈니스 로직 수행
    + 복잡성 
관심사의 분리는 추상화의 일종이다

## Service 계층의 역할

### 1.비즈니스 로직의 분리와 중앙화
도메인 객체간의 상호작용을 조정함  
Presentation Layer, Persistence Layer, Domain Model과 전부 상호작용한다

### 2. 각 계층의 역할 명확화
Controller는 사용자의 요청을 처리 생성  
Service 는 비즈니스 로직(비즈니스 로직에 대한 정의가 필요하다)  
Persistence Layer는 DB와의 상호작용을 담당  


### 3. 트랜잭션 관리
데이터의 일관성과 무결성을 보장하기 위해 트랜잭션 관리를 한다.  
여러 서비스를 통해 한 단위 기능의 원자성이 파괴되는 것을 예방  
트랜잭션은 AOP를 통해셔 구현되는데 컨트롤러에는 인터페이스 안씀


> AOP
>

### 4. 코드 재사용성 촉진
서비스 계층은 로직을 캡슐화 하므로 애플리케이션의 여러 부분에서 코드 재사용성을 촉진한다.

##  Service Layer Test
Mock를 사용해야할까?

> Mock  
> 모의 가짜 객체를 뜻한다  
> 상위 개념으로는 테스트 더블이 있다  
> 하위 개념으로는 더미 , 스텁 , 페이크 객체, 테스트 스파이 Mock 가 있으며  
> 인스턴스화 수준은 더미, 특정 상태나 모습은 스텁, 실제처럼 보이면 페이크    
> 호출 여부를 감시하고 , 나중에 호출된 기록 정보를 전달한다 - 잘안씀  
> 행위를 검증할 수 있는 목적은 Mock을 사용한다

단위 테스트는
1. 빠르고
2. 격리되고
3. 반복 가능하고
4. 자체 검증이 가능하고
5. 제품코드가 작성되기 바로 전에 작성되어야 한다(?)

## 하나의 Service 안에서 여러개의 요청이 들어올때

하나의 Service에서 여러 테이블을 통해 데이터를 다뤄야 할때  
여러 DAO / Service 중 어떤걸 여러개 써야될까.

### Service 한개 DAO 여러개 
장점 : 간단한 프로젝트에서 가장좋음  
단점 : 복잡한 기능 요구시엔 비효율적  
서비스 책임 증가

### 서비스 여러개 
컨트롤러에서 여러개의 Servicr를 의존하는 방법
트랜잭션의 원자성을 지키지 못함, 순환 참조문제 발생 -> facade 패턴

> Facade Pattern
> 여러개의 저수준 인터페이스를   
> 하나의 고수준 인터페이스로 묶어주는 패턴  
> 의존성 방향 고정 순환참조 방지
> Component Service는 여러의 Module 서비스만 의존하며
> 여러개의 모듈 서비스 조합으로 하나의 복합 기능을 담당함
> Component Service에 트랜잭션을 적용해 여러 기능을 하나의
> 트랜잭션 단위에서 동작 가능하도록 함

## MVC 패턴의 본질
MVC 패턴이란?
도메인과 View를 분리하자

### 도메인 모델이란?

+ 도메인 : 해결하고자 하는 문제 영역
+ 모메인 모델 : 특정 문제와 관련된 모든 주제의 개념모델
+ domain package : 실제 DB의 테이블과 매칭될 클래스

### 디자인 패턴
반복적으로 일어난 문제들에 대한 해결책
4가지 요소를 가진다

1. 이름을 가진다
2. 반복적으로 일어나는 문제  
3. 해법
4. 결과

View는 자주 변경된다 
MVC 패턴은 효율적인 유지 보수를 위해 뷰와 도메인의 관심사를 분리하는  
디자인 패턴임  
정해진 규칙이나 구현 방법은 없다.
